跑步是一件枯燥且疲惫的事情，却是大脑最放松的时刻。身体重复着单一的动作，大脑不用再去思考那些琐碎的事情。

在跑步的时候我常常听着舒缓的纯音乐任由思绪遨游到远方，或者是思考生活工作中遇到的问题，又或者是复习或总结最近刚学的技术。

今天跑步时偶然想到刚入大学时看到的一篇文章，过了 7，8 年至今仍有深刻的印象，于是回到家之后立马搜索出那篇文章再次阅读，不由得回想起第一次阅读这篇文章的心情。
文章链接

文章算是比较诙谐有趣，大意讲的是一个高级工程师几次面试，并且最后成功进入到自己梦想的公司 — 谷歌。在面试谷歌之前的一次面试中，有一个面试官提出了一道面试题。

问题是这样的：

假设这有一个各种字母组成的字符串，假设这还有另外一个字符串，而且这个字符串里的字母数相对少一些。从算法上讲，什么方法能最快的查出所有小字符串里的字母在大字符串里都有？

比如，如果是下面两个字符串：

String 1: ABCDEFGHLMNOPQRS

String 2: DCGSRQPOM

答案是 true，所有在 string2 里的字母 string1 也都有。如果是下面两个字符串：

String 1: ABCDEFGHLMNOPQRS

String 2: DCGSRQPOZ

答案是 false，因为第二个字符串里的 Z 字母不在第一个字符串里。

作者提出了以下三种方案但并没有使面试官满意：

  1. 轮询第二个字符串里的每个字母，看它是否同在第一个字符串里。算法时间复杂度为 O(m*n) （m，n 为两个字符串的长度）

  2. 先对两个字符串排序，然后一次轮询。排序需要 O(m log m) + O(n log n)次操作，之后的线性扫描需要 O(m+n)次操作。

  3. 将长字符串字母放入 Hashtable，然后轮短字符串看是否在表中。只需要 O(n+m)次操作。

最后面试官提出一种方案：
给每一个字母配一个素数，然后遍历长串，将字母对应素数相乘，然后轮询第二个字符串，用每个字母的素数除它，如果有余数则说明有不匹配的字母，如果整个过程中没有余数，则说明是第一个字符串的子集。

最后作者在谷歌的面试上遇到同样的问题，并最后成功用这种有趣的方案提高在谷歌面试官眼中的评价。

不能说面试官的方案是更实用的方案，但是对于刚学习编程的我来说，看到这种解决问题的思路就像一个孩子发现新玩具一样欣喜。
以至于到现在我仍然记得这一篇文章，每当看到字符串对比相关问题时第一个想到的反而就是素数相除法。

想想我们工作以后有多久没有因为写代码而开心快乐，再想想我们刚开始学习编程时用代码实现功能之后的成就感。

诚然在我们工作中常常面对的是写很多重复的业务代码，但我们仍然可以从中为自己寻找一些乐趣和挑战，比如尝试写一些工具帮我们完成那些重复的工作，或者让自己的代码更精炼和易懂，又或者对我们习以为常的问题想一想有没有其他的方案。

曾经我在开发过程中遇到一个很常见的问题：数组去重。脑海里第一反应就是遍历然后用 indexOf、include 判断唯一性，或者 ES6 的 Set。当时是果断用了 Set 这种比较优雅的方法。后来晚上回想到这里时，突然想到了 JavaScript 中的对象属性唯一性（Hashtable），立马动手写出了已知的几种方案然后测试了一下耗时，发现在数组长度在一定时，其耗时竟然最短（不过使用空间也比其他的要大），当时着实也兴奋了一下。后来网上看了一下数组去重的方案，发现别人也提出过这种方案，不过自己思考出来的印象更深刻一些。

最后说一下自己感想：

  1. 程序员注定是一个要一生保持学习的职业，但是同样基础是里面最重要的一环，只有打好地基，才能稳固。

  2. 程序员要保持思考，就像学习算法和设计模式，你遇到的问题不会正好是案例，更多的是需要你从中汲取的是解决问题的思考模式和组织代码架构的能力。

  3. 保持梦想，不忘初心。
