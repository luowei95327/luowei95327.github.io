为什么要建立知识架构

要学习一个知识最好的方法就是先建立一个对应的知识框架，然后我们在学习对应的知识点的的时候就能很快的与相关知识点进行串联和联系，方便我们理解其原理。

在我刚开始学前端的时候确实没有在脑中建立一个知识架构这样的意识，常常是遇到某一个知识点就去单独的寻找针对性的文章进行阅读和理解，当时觉得豁然开朗信心满满，在过一段时间后再想那个知识点虽然会明白大概的意思，却怎么也说不清楚。

其实这其中的真正的原因是我们将里面的知识点都拆分成了没有联系的单独的点去理解，脑海中都是零零散散的知识点，并没有真正的理解和串联起来。就像我们小时候背课文或古诗，只要弄明白了上下文的意思，就可以很快的背诵下来，但是如果不理解意思，仿佛就是由几十个认识的字组成的无序组合，背诵难度成倍上涨。

举个栗子，当我们想到JavaScript的时候是不是只想到一下的知识点

  • 数据类型;
  • 类型转换;
  • this;
  • 原型链;
  • 作用域;
  • 执行上下文;
  • 闭包;
  • 。。。

从我的理解上来看，前端可以分为html+css，JavaScript，浏览器原理及API。

html+css

html可以看做一个网页的骨架，而css则是给这个骨架披上了美丽的表皮。

html作为标记语言，我们更多注重的是各种标签的用法而不是语法，对应的我们建立架构更偏向标签功能的划分。

我们都知道 HTML 的标签可以分为很多种，head 里面的我们称为元信息类标签，诸如 title、meta、style、link、base 这些，它们用来描述文档的一些基本信息。还有一类是一些诸如 section、nav 的标签，它们在视觉表现上跟 div 并没有区别，但是各有各的适用场景，我们把它们称作语义类标签。另外一类是 img、video、audio 之类的替换型媒体类标签，用来引入外部内容，平常开发中你也会经常用到。再有就是表单类的，比如 input、button。

所以，基于这样的分类，我把标签分成下面几种。

  1. 文档元信息：通常是出现在 head 标签中的元素，包含了描述文档自身的一些信息；
  2. 语义相关：扩展了纯文本，表达文章结构、不同语言要素的标签；
  3. 链接：提供到文档内和文档外的链接；
  4. 替换型标签：引入声音、图片、视频等外部元素替换自身的一类标签；
  5. 表单：用于填写和提交信息的一类标签；
  6. 表格：表头、表尾、单元格等表格的结构。

CSS也是从语言和功能两个角度介绍，并且加上了CSS最常用的3大常用预处理器。

在语言部分，会从大到小介绍 CSS 的各种语法结构，比如 @rule、选择器、单位等等。功能部分，我们大致可以分为布局、绘制和交互类。这样分类之后发现常见的布局，语义化，动画等等都能对应到我们的css架构中。

JavaScript

但实际上JavaScript作为一门编程语言有其文法和语义，运行时也有数据结构和执行过程：用一定的词法和语法，表达一定语义，从而操作运行时。

接下来，我们又按照程序的一般规律，把运行时分为数据结构和算法部分：数据结构包含类型和实例（JavaScript 的类型系统就是它的 7 种基本类型和 7 种语言类型，实例就是它的内置对象部分）。所谓的算法，就是 JavaScript 的执行过程。

执行过程我们则需要按照从大结构到小结构的角度讲解，从最顶层的程序与模块、事件循环和微任务，到函数、再到语句级的执行。我们从粗到细地了解执行过程。实例部分，对 JavaScript 来说类似基础库，JavaScipt 的内置对象多达 150 以上，详细的资料我们都可以从MDN中仔细去查看。

浏览器的实现原理和 API

“工欲善其事，必先利其器。”

浏览器作为前端工程师最重要的工作环境，每一个前端工程师都应该了解其原理才能让其更好的服务于日常工作。

浏览器的深入从浏览器设计出发，按照解析、构建 DOM 树、计算 CSS、渲染、合成和绘制的流程来理解浏览器的工作原理。

在 API 部分，主要有：事件、DOM、CSSOM 几个部分，它们分别覆盖了交互、语义和可见效果，这是我们工作中用到的主要内容。

结语

这篇文章主要借鉴winter老师《重学前端》的课程加上自己的一些理解之后写成的。主要把前端的基础知识分为JavaScript、HTML、CSS 以及浏览器四大重点模块，每个模块也分别有自己的技术重点。

这篇文章最主要的作用就是在跟着架构的同时，我们也在梳理平常可能是零散学到的知识，不仅能够加深我们的记忆，还能让我们以后遇到问题能从体系来思考和分析问题。

当然从不同角度也能建立不同的架构，希望每一个前端工程师都能打下坚实的地基。
